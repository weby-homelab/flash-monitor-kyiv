# Правила розробки (Протокол нульової толерантності до регресій)

Цей документ описує суворі правила розробки, яких повинні дотримуватись всі розробники (і ШІ-асистенти) під час роботи над проєктами в цьому репозиторії.

## 1. Заборона «сліпих» коммітів
Перед кожним `git commit` та `git push` необхідно завжди перевіряти зміни:
```bash
git diff --cached
```
**Уважно читайте кожну строчку.** Якщо ви бачите несподівані видалення коду або зміну логіки, яку ви не планували робити, негайно зупиніться і скасуйте зміни. Заборонено сліпо приймати конфлікти злиття (merge conflicts) за допомогою `git checkout --theirs` або `git checkout --ours` для Python файлів або файлів логіки. Ви зобов'язані відкрити файл і вирішити конфлікт вручну.

## 2. Стратегія роботи з гілками (Branching)
Проєкт використовує дві основні гілки:
- `main` - для Docker-середовища (HTZNR-TEST).
- `classic` - для Bare-Metal середовища (HTZNR).

**Ніколи не вносьте зміни безпосередньо в `main` або `classic`.** 
- Використовуйте тимчасові гілки (`fix/...`, `feat/...`, `docs/...`).
- Перед тим як злити тимчасову гілку в `main` або `classic`, завжди переконайтеся, що `main` та `classic` синхронізовані між собою. 
- Ніколи не допускайте ситуації, коли одна з головних гілок містить оновлення, яких немає в іншій (розсинхронізація версій або коду).

## 3. Ізоляція змін
- Якщо завдання полягає в оновленні документації (`README.md`), додавайте в комміт **тільки** файли документації.
- Заборонено використовувати `git add .`, якщо ви не впевнені абсолютно в усіх змінених файлах. 

## 4. Обов'язкове тестування (Unit-тести)
- Жодна зміна в логіці роботи скриптів (генерація звітів, розрахунки часу) не може бути задеплоєна без покриття автоматичними тестами.
- Тести знаходяться в директорії `tests/`.
- Перед пушем обов'язково запускайте перевірку:
```bash
python3 -m pytest tests/
```
Якщо тести не проходять, пуш заборонено!

## 5. Чек-лист перед релізом
1. Перевірити тести (`pytest`).
2. Перевірити синхронізацію гілок `main` та `classic`.
3. Оновити файл `VERSION`.
4. Оновити версію в `docker-compose.yml`, `README.md` та `README_ENG.md`.
5. Зробити `git tag vX.Y.Z` і відправити код на сервер.
